Concept-oriented programming

From Wikipedia, the free encyclopedia

Jump to: navigation, search
Not to be confused with concept programming.

Concept-oriented programming is a programming paradigm which uses concepts as the main programming construct. One of the main assumptions of this approach to programming is that any system can be viewed as consisting of two types of functionality:

  • Business methods (BMs) which are used explicitly in the source code, and
  • Representation and access (RA) methods which are called implicitly when BMs are being called.

RA functions cross-cut the whole program and are dual (orthogonal) to BMs. In other words, as we use BMs by calling class methods something always happens behind the scenes. For example, if a Java method is called then the Java Virtual Machine (JVM) needs to resolve the target object reference in order to execute it. Furthermore, when the Java reference has been resolved the obtained memory handle might need to be also resolved into a physical address. This procedure is nested and cross-cuts many levels where very different functions can be executed, for example, security checks, locks, access counting etc. This level of the system functionality is hidden because its functions are executed implicitly as objects are being accessed.

In the concept-oriented programming it is assumed that RA functions account for a great deal (and even most) of the system complexity. Thus there is a need in describing these hidden RA functions as an integral part of the program. An important design goal consists in retaining the illusion of instant access providing at the same time facilities for specifying what happens behind the scene. For example, assume that we have class A and want to call its business method: A o; o.myMethod();. The idea of COP is that the object o can reside anywhere in the world while this line of code will be the same. It is the task of the environment to provide appropriate RA functions to access it while the compiler has to activate them implicitly. For example, the target object o can be in main memory where it is represented by a memory handle, in JVM where it is represented by a Java reference, on another computer where it is represented by a remote reference, or on Mars where it is represented by some kind of global identifier. We do not want and do not need to know where the object is really stored and how it has to be accessed in order to work with this object by calling its BMs.

In order to describe such a system where the two types of functionality co-exist, the new approach uses concepts as the main programming construct. A concept consists of one object class and one reference class which provide the facilities for describing the two parts of the system functionality. The idea is that we use concepts instead of classes by providing a concept hierarchy where each concept is included in some parent concept. Then intermediate parent concepts provide the necessary RA functions for the child concepts. The program execution then is viewed as an indirect representation and access to child objects via their parent objects playing a role of an intermediate environment. One concept can be thought of as a space, and then any BM call has to pass through a sequence of space borders each automatically triggering some intermediate actions. The target BM is only the very last step in this sequence while a significant part of functionality is concentrated on intermediate space borders. One important property and design goal of the concept-oriented programming is that it makes it possible to describe the whole sequence of actions (RA functions) rather than only the last step (BMs).

Depending on the parent concepts, the target concept will be represented and accessed in one or another way. For example, if we know that concept or class A produces small objects which are frequently created and deleted then we might want to develop a special container for them and then include concept A in it: A in ContainerForTinyObjects {...}. If later we want to make these object persistent then this target concept can be included in the corresponding parent concept: A in ContainerForRemoteObjects {...}. Notice that these parent concepts are developed as an integral part of the concept-oriented program. A concept-oriented program uses concepts instead of classes and concept inclusion relation instead of inheritance. The difference is that objects are represented by means of references described by parent concepts and accessed via methods provided in parent concepts.
